# دليل تدوير الأسرار الأمنية والمفاتيح البرمجية (Secret Rotation Runbook)
## نظام إدارة الموارد المؤسسية: ApexSaaS ERP

يوضح هذا الدليل التشغيلي كيفية إجراء تدوير أمني دوري أو طارئ للمفاتيح السرية ورموز التشفير الخاصة بنظام **ApexSaaS ERP** لضمان استمرارية التشغيل والنزاهة الأمنية دون التأثير على تجربة المستخدمين النشطين في المنصة.

---

### 1. مقدمة ونطاق التدوير (Scope of Rotation)

يعتمد النظام على مجموعة من الأسرار الحرجة التي يجب تدويرها بشكل دوري (كل 90 يوماً كحد أقصى) أو بشكل فوري في حال الاشتباه في تسرب أمني:

1. **`DATABASE_URL`:** رابط الاتصال المباشر بقاعدة بيانات PostgreSQL الإنتاجية.
2. **`JWT_SECRET`:** المفتاح السري المستخدم لتوقيع والتحقق من رموز الوصول قصيرة الأجل (`accessToken` - صلاحية 15 دقيقة).
3. **`JWT_REFRESH_SECRET`:** المفتاح السري المخصص لتوقيع والتحقق من رموز التحديث طويلة الأجل (`refreshToken` - صلاحية 7 أيام).
4. **`GEMINI_API_KEY`:** رمز الاتصال بمحرك الذكاء الاصطناعي التوليدي الآمن للتحليل المالي المتقدم.

---

### 2. معالجة تدوير مفاتيح الـ JWT دون إخراج المستخدمين (Zero-Disruption JWT Rotation)

لضمان عدم خروج المستخدمين النشطين فجأة عند تغيير مفاتيح الـ JWT، يدعم النظام آلية التحديث الصامت والتلقائي:
* **صلاحية الـ `accessToken` قصيرة جداً (15 دقيقة)**، بينما الـ **`refreshToken` يمتد لـ 7 أيام**.
* عند تدوير `JWT_SECRET` الخاص بالـ `accessToken`:
  1. تنتهي صلاحية كافة الـ `accessTokens` القديمة فوراً أو تفشل في التوقيع الجديد.
  2. يقوم جدار الحماية الأمامي بمقاطعة الخطأ (HTTP 401) وتمرير الـ `refreshToken` بشكل صامت وموازٍ لطلب مفتاح جديد عبر معالج `rotateRefreshToken`.
  3. يتم منح المستخدم `accessToken` جديد تماماً موقع بالمفتاح السري الجديد دون أن يلاحظ أي انقطاع في جلسته التشغيلية.
  4. يتم تدوير الـ `refreshToken` نفسه عبر آلية (Refresh Token Rotation - RTR) لمنع هجمات إعادة التشغيل (Replay Attacks).

---

### 3. التدوير التلقائي في بيئة Kubernetes

تم توفير سكربت مؤتمت بالكامل لإجراء تدوير آمن بنقرة واحدة وتحديث أسرار الـ Pods دون أي فترة توقف (Zero-Downtime Rolling Update).

#### خطوات التشغيل:
1. تأكد من توجيه بيئة الـ CLI الخاصة بك إلى العنقود الصحيح (Cluster context) وضبط اسم الـ Namespace الافتراضي.
2. قم بتشغيل السكربت المخصص:
   ```bash
   ./kubernetes/secret-rotation.sh
   ```

#### ما يقوم به السكربت فعلياً:
* توليد مفاتيح تشفير عشوائية عالية الإنتروبيا (High Entropy) باستخدام `openssl rand -hex 32`.
* ترميز القيم الناتجة بنظام Base64 بشكل آمن ومطابقتها لمواصفات هياكل Kubernetes.
* تطبيق التحديث مباشرة على كائن الـ Secret المسمى `erp-secrets`.
* إطلاق عملية إعادة بناء تدريجية (Rolling Update) للـ Deployment المسمى `apexsaas-erp-deployment` لتبديل الحاويات بالتدريج دون انقطاع الخدمة عن أي مستخدم نشط.

---

### 4. التدوير اليدوي خطوة بخطوة (Manual Emergency Rotation)

في حال عدم توافر بيئة أتمتة كاملة أو للتدوير اليدوي، اتبع الخطوات التالية بدقة:

#### الخطوة 1: توليد المفاتيح السرية الجديدة
قم بتوليد سلاسل نصية مشفرة بطول أمان لا يقل عن 32 حرفاً:
```bash
# توليد مفتاح توقيع JWT الجديد
openssl rand -hex 32

# توليد مفتاح التحديث للـ JWT
openssl rand -hex 32
```

#### الخطوة 2: تحديث قيم الأسرار في Kubernetes يدوياً
قم بتحويل القيم الجديدة إلى Base64:
```bash
echo -n "المفتاح_الجديد_هنا" | base64
```
ثم قم بتعديل ملف الـ Secrets الخاص بـ Kubernetes وتطبيق التعديل:
```bash
kubectl edit secret erp-secrets -n production
```
استبدل قيم الحقول `jwt-secret` و `database-url` بالقيم المرمزة الجديدة.

#### الخطوة 3: تحديث متغيرات البيئة المحلية (On-Premise / PM2 / Docker Compose)
إذا كنت تستخدم Docker Compose أو تخدم يدوي، قم بتحديث ملف `.env` التشغيلي:
```env
JWT_SECRET=المفتاح_الجديد_الذي_تم_توليده_أعلاه
JWT_REFRESH_SECRET=مفتاح_التحديث_الجديد_أعلاه
DATABASE_URL=رابط_الاتصال_الجديد_بالقاعدة
```

#### الخطوة 4: إعادة تشغيل تطبيق الخادم بالتدريج
قم بعمل ريستارت للخادم لضمان تحميل المفاتيح الجديدة في الذاكرة دون قتل الجلسات الحالية:
* **في Kubernetes:**
  ```bash
  kubectl rollout restart deployment/apexsaas-erp-deployment -n production
  ```
* **في Docker Compose:**
  ```bash
  docker compose up -d --no-deps --build erp-app
  ```

---

### 5. التحقق والتأكد من سلامة التدوير (Verification & Logging)

1. **مراقبة السجلات (Application Logs):**
   تأكد من عدم وجود أخطاء `JWT signature failure` غير طبيعية في نظام مراقبة الأخطاء.
2. **سجلات التدقيق الأمني:**
   يتم تسجيل عملية تدوير الأسرار في سجلات التدقيق الأمني العامة للنظام مع توضيح وقت وتاريخ العملية ومعرّف الجلسة التشغيلية لضمان الامتثال لمتطلبات آيزو 27001 ومعايير الحوكمة المالية.
3. **معدل فشل المصادقة:**
   تأكد من بقاء منبهات Prometheus الخاصة بمعدل فشل تسجيل الدخول (`HighAuthFailureRate`) تحت حدود التحذير الآمنة بعد تدوير المفاتيح مباشرة.
